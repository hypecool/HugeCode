# Local Figma Bridge

This is the repository's free, stable Figma workflow.

It exports selected Figma nodes as official `JSON_REST_V1` data, writes preview assets for local
inspection, and includes a regular-plugin token inspector that maps bound Figma variables back to
repo token paths, CSS variables, and `vanilla-extract` contract access. It does not require Figma
MCP, Dev Mode, or a paid seat for normal local use.

## What it does

- exports a selected Figma node through the regular Plugin API
- writes companion `PNG` and `SVG` previews from the same selected node
- resolves bound Figma variable names against the checked-in `figmaCodegenMap` generated by the
  design-system token pipeline
- shows repo-native token snippets in a regular Figma plugin
- posts exported JSON to `http://127.0.0.1:3845/figma-export` for local tooling
- writes captured exports to `.figma-workflow/figma-exports`
- retains only the newest export bundle per Figma selection and the newest three bundles overall by default
- generates a `.summary.json` file with node counts, colors, sample layer names, and image refs
- generates a `.manifest.json` file that anchors the staged Figma pipeline

## Files

- `manifest.json`: import this into Figma Desktop
- `code.js`: plugin main thread
- `ui.html`: plugin UI with export actions plus the free token/code inspector
- `receiver.mjs`: local HTTP receiver for exported JSON
- `fetch.mjs`: optional headless registry-driven REST fetcher for maintenance workflows
- `resolve.mjs`: link-first local resolver that prefers cached or locally materialized artifacts before any explicit REST fallback
- `inspect.mjs`: print a summary for the latest or specified export JSON
- `smoke.mjs`: local smoke test for the receiver path, the mocked fetch path, and the plugin UI workflow

## Canonical local workflow

Prepare the free plugin assets:

```bash
pnpm -C tools/figma bridge:prepare
```

Start the receiver:

```bash
pnpm -C tools/figma bridge:listen
```

In Figma Desktop, import:

`scripts/figma-json-bridge/manifest.json`

Then:

1. Open the target file.
2. Select exactly one node.
3. Run `HugeCode Local Figma Bridge`.
4. For token/code lookup, review `Token And Code Inspector`.
5. For export artifacts, click `Export selection or target`.
6. Click `Send last export to localhost`.

Inspect the latest export:

```bash
pnpm -C tools/figma bridge:inspect
```

Resolve a Figma link into the best local artifact path for agent work:

```bash
pnpm -C tools/figma bridge:resolve --url "https://www.figma.com/design/..."
```

This command is the preferred link-first entrypoint for agents and free-account workflows. It:

- reuses an exact local artifact for the requested node when available
- materializes the requested subtree from a matching page-scale source export when possible
- only falls back to REST when `--allow-fetch` is explicitly provided

Run the governed downstream pipeline:

```bash
pnpm -C tools/figma pipeline:production
```

## Token inspector behavior

The token inspector is the default path for local design-token lookup.

It currently supports:

- exact variable-name matches
- normalized suffix matches
- manual search by token path, CSS variable, or contract access

It intentionally does not guess from raw color values. If the current selection has no bound
variables or no matching repo token path, the UI shows `Unmapped`.

## Validation

Run:

```bash
pnpm -C tools/figma bridge:smoke
```

This validates:

- localhost receiver writes a JSON artifact bundle
- mocked headless fetch path still works for maintenance tooling
- Playwright UI workflow covers mapped selection, manual search, and `Unmapped` output

To refresh the generated token/code map first:

```bash
pnpm tokens:build
```

## Real Figma validation steps

1. Run `pnpm -C tools/figma bridge:prepare`.
2. Start `pnpm -C tools/figma bridge:listen`.
3. Import `scripts/figma-json-bridge/manifest.json` into Figma Desktop.
4. Open your duplicated Figma file.
5. Select the target node.
6. Run `HugeCode Local Figma Bridge`.
7. Verify token lookup in `Token And Code Inspector`.
8. Export and send the artifact to localhost when you need pipeline input.

If successful, the exported files will be written under `.figma-workflow/figma-exports`:

- `*.json`: full exported payload
- `*.png`: rendered preview
- `*.svg`: vector preview
- `*.summary.json`: lightweight resource summary
- `*.manifest.json`: source, asset, and checksum manifest for downstream pipeline stages

Old export bundles are pruned automatically after each write so repeated local runs do not grow
`.figma-workflow/figma-exports` without bound. If you need a different retention window, override
`FIGMA_EXPORT_MAX_BUNDLES` and `FIGMA_EXPORT_MAX_BUNDLES_PER_SELECTION`.

## Staged pipeline

After a raw export bundle exists, generate governed downstream artifacts with:

```bash
pnpm -C tools/figma pipeline:production
```

This either writes audit-only staged artifacts or the full develop flow, depending on export scope.
Use `pnpm -C tools/figma pipeline:run` only when you intentionally want the broader audit/token pipeline
without production gating.

The production workflow writes:

- `*.classified-node-graph.json`
- `*.primitive-tokens.json`
- `*.semantic-tokens.json`
- `*.component-inventory.json`
- `*.qa-report.json`
- codegen and promotion review artifacts only when the selected node is safe for governed code generation

For free-account local development, prefer focused-family iteration from an existing page export:

```bash
node scripts/figma-pipeline/fetch-focus-batch.mjs \
  --plan docs/design-system/figma-focus-plan.linear-dark-mode.json \
  --families Button,Input,Select
```

This reuses the matching root export to materialize child-node artifacts locally before it falls
back to any live Figma request.

For arbitrary Figma links that are not already part of a checked-in focus plan, use:

```bash
pnpm -C tools/figma bridge:resolve --url "https://www.figma.com/design/..."
```

This keeps link-driven agent work on the free local path and avoids browser-canvas scraping.

## Optional maintenance-only fetch path

These commands remain available for controlled maintenance work, not for the normal local loop:

```bash
pnpm -C tools/figma bridge:doctor --resource linear-ui-community-copy-card-reference
FIGMA_ACCESS_TOKEN=... pnpm -C tools/figma bridge:fetch --resource linear-ui-community-copy-card-reference
```

Use them only when you intentionally need a fresh REST snapshot for a registry-backed node and the
local plugin export or artifact cache is insufficient.

## Production notes

- `doctor.mjs` checks target resolution, token presence, and whether the configured Figma API host is reachable before a headless fetch.
- `inspect.mjs` emits a `workflowRecommendation` block so page-scale exports stay in audit mode by default.
- `fetch.mjs` reuses recent local artifacts for the same node by default and performs bounded retries when the Figma API returns `429`.
- `fetch.mjs` fails fast when Figma returns an extremely large `Retry-After`, so token-based maintenance paths do not sleep indefinitely.
- raw artifact manifests retain Figma REST document metadata when available, including document name, version, and last modified time
- treat the Desktop plugin path as the default local handoff path whenever network access, account access, or API quotas are uncertain

## Notes

- `fetch.mjs` uses the official Figma REST API to write the same local artifact bundle shape as the plugin path.
- `smoke.mjs` covers the receiver POST path, the mocked headless REST fetch path, and the plugin UI workflow with structured payload assertions.
- the manifest uses `devAllowedDomains` for `http://127.0.0.1:3845` and `http://localhost:3845`
- if you need a different localhost port, update both `manifest.json` and the receiver command
- imported local manifests do not need a published plugin id
